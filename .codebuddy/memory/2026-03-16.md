# 2026-03-16 - updated_at字段验证错误修复

## 问题描述
2026-03-16 18:29:34，系统出现FastAPI响应验证错误：
```
1 validation errors:
  {'type': 'datetime_type', 'loc': ('response', 0, 'updated_at'), 'msg': 'Input should be a valid datetime', 'input': None}
```
错误发生在 `/api/v1/workspaces/` 端点，当数据库中的 `updated_at` 字段为 NULL 时。

## 根本原因
1. **SQLAlchemy模型定义**：
   ```python
   updated_at = Column(DateTime(timezone=True), onupdate=func.now())
   ```
   `onupdate=func.now()` 只在记录更新时设置时间，新创建的记录该字段为 NULL。

2. **Pydantic模型定义**：
   ```python
   updated_at: datetime  # 必须的datetime类型
   ```
   当数据库返回 NULL 值时，Pydantic 验证失败。

## 修复方案
将所有可能为 NULL 的 `updated_at` 字段改为可选类型：

1. **workspace.py** (`backend/app/schemas/workspace.py`)：
   ```python
   updated_at: Optional[datetime] = None  # 改为可选
   ```

2. **user.py** (`backend/app/schemas/user.py`)：
   - `RoleResponse` 模型中的 `updated_at` 字段
   - `UserResponse` 模型中的 `updated_at` 字段
   都改为 `Optional[datetime] = None`

## 验证
1. 检查了所有其他 Pydantic schemas，发现它们已经正确处理了 `updated_at` 字段为可选类型。
2. 检查了相关 API 端点，确保它们都使用 Pydantic schemas 而不是直接使用 SQLAlchemy 模型。
3. 运行 linter 检查，没有语法错误。

## 注意事项
1. `created_at` 字段使用 `server_default=func.now()`，数据库会自动设置，通常不会为 NULL。
2. 类似地，其他使用 `onupdate=func.now()` 的字段在相应 Pydantic schemas 中都应定义为可选类型。
3. 修复后，前端代码应能处理 `updated_at` 可能为 `null` 的情况。
